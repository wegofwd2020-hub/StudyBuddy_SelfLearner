# Quality & Compliance Gates — how we curate content

> **Purpose.** A single source of truth for the gates every piece of generated
> content passes through, *and* how those gates are surfaced — to the reader, to a
> compliance reviewer/auditor, and on the product surface. The gates are scattered
> across six ADRs and three runtimes; this doc consolidates them so "what do we
> enforce, and where do we show it?" has one answer.
>
> **Why now.** Two recent shifts make this legibility work load-bearing:
> 1. **The LLM seam is a shared package** — `wegofwd-llm` (ADR-012), consumed by
>    both Mentible *and* Pramana. The validation + provenance gates are now
>    *package-owned*, so the **same gate runs identically regardless of producer**.
> 2. **Multi-provider is real** (ADR-005/012/014). Content can be generated by any
>    of N models. The gates are what let us say "any model" without "inconsistent,
>    untraceable results."

---

## 1. The gates (inventory)

Ordered as content flows: machine validation → provenance → integrity → human
approval → lifecycle. **Status** is honest about what is *wired* vs *specced*.

| # | Gate | What it enforces | Lives in | Status |
|---|------|------------------|----------|--------|
| 1 | **Schema validation + repair** | Every model response is parsed and validated against the output schema; on failure the `generate_validated` conformance loop repairs up to `max_repairs`, then fails the job. Transport errors (timeout/refusal) fail fast — they are not schema failures. | `wegofwd_llm.conformance.generate_validated`; wired at `backend/src/generate/tasks.py` | **Active** |
| 2 | **Generation provenance** | Each result is stamped with `provider`, `model` (+ config fingerprint/versions) so we always know *which LLM under which config* produced it. | `wegofwd_llm.registry.provenance(...)`; stamped onto the result payload in `tasks.py`; `pipeline/providers/config.py` resolves the effective per-book/per-level choice | **Active** |
| 3 | **Format-drift heuristics** | Sections whose title is tabular-by-nature (Balance Sheet, truth table…) must contain a GFM table; formula sections must contain KaTeX. Non-fatal warnings → review queue / early prompt-drift signal. | `pipeline/content_format_validator.py`, wired at `backend/src/generate/tasks.py` (`_format_warnings`); warnings ride on the job status row | **Active** |
| 4 | **Traceability / citations** | Every claim cited to a source clause; `source_definitions` + per-module `citations`; `every_claim_cited` constraint on the Package Request. | ADR-011 §4 (Consumable Package manifest) | **Spec** (Pramana handoff) |
| 5 | **Integrity / tamper-evidence** | `content_hash` + `signature` on the package, verified on ingest; mismatch → quarantine, never silently published. | ADR-011 §6 | **Spec** (Pramana handoff) |
| 6 | **Human approval (SoD)** | Generated content is an *untrusted draft* until a qualified human (≠ the generator) approves it with attestation; only then is it published to an immutable `CourseVersion`. **Invariant across producers**: Mentible-pushed *and* Pramana-in-process content enter the **same** `ContentDraft → IN_REVIEW → APPROVED → PUBLISHED` gate (ADR-013 D4). | ADR-011 §7, ADR-013 D4 (Pramana side) | **Spec / partly built (pramana PR #1)** |
| 7 | **Release lifecycle + watermark** | DRAFT watermark (PDF) / "not for distribution" notice (EPUB) until release; release stamps edition + colophon; versioned artifacts. | ADR-008 | **Spec** |

**The shape that matters:** gates 1–3 are **machine, in-Mentible**; gates 4–6 are
**provenance + human, at the Pramana boundary**; gate 7 is **presentation**. A
fully honest "what we enforce" story spans both products — which is exactly why
the surfaces in §4 need to make that boundary invisible to the reader.

## 2. Multi-provider: presentation uniformity + provenance disclosure

Multi-provider splits "how we present information" into two distinct guarantees.

### 2a. Presentation *uniformity* — the gates make the model invisible
A reader must **not** be able to tell from the formatting which model wrote a
page: same tables, same KaTeX, same structure, whatever the backend. That
uniformity *is* the product guarantee, and gates **1** (schema → identical shape)
and **3** (format-drift → catches a weaker model that passed schema but skipped
the table/formula) are what deliver it. ADR-005's open question is the live risk:
*non-Anthropic models vary in JSON instruction-following* — so with N providers,
gate 3's per-provider warning rate becomes a **presentation-consistency metric**,
not just an admin nicety.

### 2b. Provenance *disclosure* — which model wrote this, shown honestly
Provenance is captured (gate 2) but must be *shown*. Two complications are now
real:

- **A book can mix models across levels** (`pipeline/providers/config.py`: a book
  narrows the author's allow-set; a level may pin a different provider/model). So
  a single "Generated by X" colophon line can be a lie.
- **A provider's output quality varies**, and we already say so in shipped code:
  the mobile BYOK picker tags each provider `tier: "authoring" | "experimental"`
  (`mobile/src/constants/providers.ts`) — Anthropic is `authoring`; OpenAI/Groq/etc.
  are `experimental` with the note *"treat output as draft-grade."* This is the
  first place we present a quality signal about the model to the user; it should
  stay consistent with the colophon/report below.

**Disclosure policy (recommended):**

| Surface | What it shows | Why |
|---|---|---|
| In-app picker | per-provider `tier` + honest note | sets expectations *before* generation |
| Artifact **colophon** | summarized ("frontier models, all human-reviewed") or single model if the edition used one | clean for the reader; never claims a single model when mixed |
| **Compliance Report** (manifest render) | full per-module `provider` + `model` + `content_hash` | auditors want the mix; learners don't |

## 3. The gate trail (one-liner for UI / marketing)

> **Validated → Provenance-stamped → Cited → Signed → Human-approved → Published**

Each step maps 1:1 to a row in §1 (1 → 2 → 4 → 5 → 6 → 7). Use this named trail
verbatim on the landing page and as an in-app badge so the public language and the
engineering reality stay in lockstep.

---

## 4. Where each gate is surfaced (by audience)

1. **The reader — the artifact colophon.** Already plumbed for the lifecycle
   stamp (ADR-008 `colophon.ts`). Extend it into the single legible trust surface:
   model/provenance summary (per §2b), generation date, short `content_hash`,
   frameworks/clauses covered, edition/version, and "Human-approved on <date> by
   <role> — approver ≠ author." Content change, not new infra.
2. **The compliance SME / auditor — a generated Compliance Report.** The ADR-011
   manifest already carries provenance, citations, hash, signature — but it is a
   wire format. Render it as one human artifact per package: clause-coverage
   matrix, every-claim-cited checklist, the approval attestation, the model mix,
   and the audit chain. This is the sales differentiator and the audit deliverable
   in one.
3. **The buyer / product surface — the gate trail (§3) + provider tier (§2b).**

## 5. Honest gaps (do not let the doc over-claim)

- **Gate 3 fires only on the lesson surface so far.** It is now wired into
  `tasks.py` (`_format_warnings`, after gate 1) and its warnings ride on the job
  status row + a per-provider `format_warnings` log line. But it runs on the
  lesson generator only; the book/tutorial and Pramana-package surfaces still need
  the same call at their write points. The validator's lesson path is a no-op
  upstream, so we adapt lesson sections (`heading`/`body_markdown`) to the
  tutorial shape it checks rather than edit the vendored file (ADR-002).
- **Experimental providers are unmeasured.** The `experimental` tier providers
  have no measured conformance tier yet (Phase 5). Until then their output is
  honestly draft-grade — the picker says so; the colophon/report must not imply
  otherwise.
- **Gates 4–7 are Pramana-side spec.** ADR-011 (Proposed) and ADR-014 (Proposed)
  are not yet built end-to-end; ADR-013 (Accepted) fixes the invariant approval
  gate but the ingestion/approval machinery is pramana-side (PR #1). Any external
  "we enforce" claim should scope to what is live in this repo (gates 1–2) plus
  what the handoff *contract* guarantees, not assert the full chain is shipped.
